Two worlds. One gap.

OLTP, where data is born
  • Postgres, row-store: insert one order fast
  • B-tree point lookups, transactions, constraints
  • Terrible at “scan 10M rows and aggregate”
OLAP, where questions get answered
  • DuckDB, column-store: scan + aggregate fast
  • Vectorized, compressed, 10×+ on analytics
  • Terrible at single-row inserts & lookups
Your data is born in OLTP but analytics needs it in OLAP. Something has to move it across the gap, reliably, every day. That something is a data pipeline, today, the classical batch kind.

Vocabulary

Where does the T happen?

  • ETL: extract → transform → load. The shape every textbook draws.
  • ELT: extract → load → transform. The shape the cloud warehouse era made cheap.
Our pipeline is a hybrid. We extract with psycopg, then transform and load inside DuckDB: ELT-style performance from a technically-ETL shape. The label matters less than the four problems, which are universal.

ETL vs ELT · a short history

Load-first came first.

The order keeps flipping with the cost of compute and storage. Three eras, and the reason each one won:

EraShapeHow it workedWhy it won
1990sELT (before the term)Load flat files to staging, transform in the warehouseOnly option; staging was expensive, deleted after use
2000sETLAn integration server (Informatica, etc.) transforms in memory before loadKilled staging contention; visual drag-and-drop over SQL
2010s+ELTCheap cloud storage; transform in-warehouse with dbt (SQL + Jinja)Compute / storage split, separate clusters, raw data stays
Counterintuitive, but real: load-then-transform predates the dedicated ETL tools most people assume came first. The cloud didn’t invent ELT, it made it cheap again. The pendulum tracks one thing: where compute is cheapest to run.

Extract & transform

Pull a day. Aggregate it.

The obvious way: read one day of orders out of Postgres, hand the rows to DuckDB, let it do the GROUP BY. Boring on purpose, every step stays visible.

pipeline_naive.py, extract() + transform()
# EXTRACT: one day, explicit columns (never SELECT *) rows = pg.execute("SELECT id, customer_id, amount, status, created_at " "FROM orders WHERE created_at::date = %s", [day]) # TRANSFORM: push the GROUP BY into DuckDB agg = duck.execute("SELECT ?::DATE, status, SUM(amount), COUNT(*), " "ROUND(AVG(amount),2) FROM raw GROUP BY status", [day])
We pull the day into Python so each stage is legible. The frameworks (later) stage via Parquet instead, same three stages. The one thing that carries over unchanged is the load. That’s where all four problems live.
EXTRACT · TRANSFORMpsycopg → DuckDB GROUP BY
IdempotentAn operation you can apply many times with the same result as applying it once. f(f(x)) = f(x).
Problem 01 · Idempotency

Run it three times.
Same state.

The scheduler will re-run your job, retries, backfills, a nervous engineer hitting the button. The naive load appends blindly: three runs, triple the rows, and a revenue number that quietly lies.

IDEMPOTENCYsame inputs → same state
Partition replacementDelete the slice you’re about to rewrite, then insert it fresh. Re-running just rewrites the same slice.
Fix 01a

Strategy 1, DELETE + INSERT

pipeline_idempotent.py
con.execute("BEGIN TRANSACTION") try: con.execute("DELETE FROM daily_revenue WHERE date = ?", [day]) for r in rows: con.execute("INSERT INTO daily_revenue VALUES (?,?,?,?,?,?)", r) con.execute("COMMIT") # both steps land together… except Exception: con.execute("ROLLBACK"); raise # …or neither does
Works without a primary key. You define the “partition” (here, one date) and replace it wholesale. The most robust default.
IDEMPOTENT LOADdelete-insert
UPSERTInsert, but on a key collision overwrite the existing row. One statement, no explicit delete.
Fix 01b

Strategy 2, UPSERT

pipeline_idempotent.py, load_upsert
con.execute(""" INSERT OR REPLACE INTO daily_revenue_keyed (date, status, total_revenue, order_count, avg_order_value, loaded_at) VALUES (?, ?, ?, ?, ?, ?) """, row) # re-inserting the same key overwrites in place
$ uv run src/pipeline_idempotent.py 2024-01-15 --strategy upsert
UPSERT needs a key. It targets daily_revenue_keyed, which declares PRIMARY KEY (date, status), the conflict target. No key, no “replace.”
UPSERT only touches keys that reappear. If a status drops out of a day’s data, its stale row survives, an orphaned key. DELETE + INSERT rewrites the whole partition, so it clears vanished keys too. That’s why it’s the safer default; experiment_upsert_gap.py proves the difference.
IDEMPOTENT LOADinsert-or-replace · orphaned keys
Problem 01 · live

The naive load grows.

The load is a blind INSERT loop, no DELETE, no key. Run the same day three times and watch the proof harness:

$ uv run python src/prove_idempotent.py --loader naive
run rows checksum 1 3 b04a42b3c882 2 6 da58b0d4b47f 3 9 593f71c60c10 NOT IDEMPOTENT: 3 distinct states across 3 runs.
Every retry appends another full day. The checksum changes because the revenue triples, a number that quietly lies.
NAIVE LOADblind INSERT → duplicates
Fix 01 · live

Same inputs. Same state.

Swap the blind INSERT for the DELETE + INSERT in one transaction. Re-run the same three times:

$ uv run python src/prove_idempotent.py 2024-01-15
run rows checksum 1 3 b04a42b3c882 2 3 b04a42b3c882 3 3 b04a42b3c882 IDEMPOTENT: all 3 runs produced identical state (3 rows).
Identical checksum, every run. The DELETE makes a re-run a no-op, the partition is replaced with the same bytes. This is the deliverable standard: prove_idempotent.py.
IDEMPOTENT LOADidentical checksum ×3
Problem 02 · Recovery

The trick is the transaction.

DELETE then INSERT looks safe, until it crashes after the delete. Without a transaction, you’ve just emptied yesterday’s revenue and written nothing back. That’s the crash-mid-load failure, and the transaction is how you recover from it.

DELETE, every INSERT, and (later) the watermark write share one transaction. Either the whole new state commits, or the old state survives untouched. That is what makes a re-run safe.
ATOMICITYall-or-nothing
Problem 02 · live

Crash mid-load. Retry. No corruption.

Inject a random crash inside the INSERT loop, wrap the load in a retry. Because the load is idempotent, retrying is safe, not a duplication machine.

$ uv run python src/pipeline_failure.py --fail-prob 0.4
Attempt 1/8 FAILURE: crash after 1 of 3 rows, rolled back, target unchanged Attempt 2/8 FAILURE: crash after 2 of 3 rows, rolled back, target unchanged Attempt 3/8 Success on attempt 3 (3 rows)
Two crashes, zero corruption. Each failed attempt ROLLBACKs the DELETE too, the target is byte-for-byte its old self until a clean COMMIT. Retries without idempotency just duplicate faster.
FAILURE RECOVERYrollback + retry
WatermarkA durable record of which partitions finished. A re-run skips them; a backfill fills the gaps. (Not the event-time watermark of streaming: same word, different idea. That one’s in L5.)
Problem 02 · Recovery, at range scale

Load a range. Skip what’s done.

Recovery scales from one date to a whole range: a pipeline_metadata ledger records each finished date, so a crash partway through a backfill just resumes. Run the same range twice:

run 1 · 01-10 .. 01-20
2024-01-10 loaded 3 (11,112) 2024-01-11 loaded 3 (11,032) … … 2024-01-20 loaded 3 (11,239)
run 2 · same range
2024-01-10 skip (watermark) 2024-01-11 skip (watermark) … … 2024-01-20 skip (watermark)
11 dates loaded, then 11 skipped. --force reprocesses anyway, and that’s safe, because the load underneath is idempotent.
WATERMARKload once · skip · backfill gaps
The subtlety that bites

“Loaded” and “recorded as loaded” are one fact.

pipeline_watermark.py, load_and_mark()
con.execute("BEGIN TRANSACTION") con.execute("DELETE FROM daily_revenue WHERE date = ?", [day]) # ... INSERT the day's rows ... con.execute("INSERT OR REPLACE INTO pipeline_metadata ...") # the mark con.execute("COMMIT") # data + watermark land together, or not at all
Mark after a separate commit → crash in between → date looks unloaded, silently reloads. Mark before the data commits → date looks done but has no data. Both are silent corruption. One transaction erases the gap, exactly what an orchestrator’s metadata DB does for you.
ATOMIC WATERMARKdata + metadata in one txn
DimensionDescriptive context for facts, who/where/what. Unlike facts, dimensions mutate over time.
Problem 03 · Slowly Changing Dimensions

The dimension that won’t sit still.

Facts are immutable, an order happened, for $42, on the 15th. But the customer behind it changes: they move from Chicago to LA.

  • Overwrite it, and every past order suddenly looks like it came from LA. History rewritten.
  • Ignore it, and new orders are tagged with a stale city. Present wrong.
  • You need to answer “where did this customer live when the order happened?”
SCDdimensions mutate
Three ways to remember

Type 1, 2, 3.

TypeStrategyHistoryCost
Type 1Overwrite in placeNone, past is rewrittenCheapest
Type 2New row + validity windowFull, every version keptMore rows, more logic
Type 3“previous value” columnLast change onlyLimited, awkward
Type 2 is the workhorse. It’s the only one that answers “as of when?”, so it’s what we build. The price: your dimension grows a row every time an attribute changes.
SCD TYPES1 overwrite · 2 version · 3 prior-col

Problem 04 · Schema drift

Upstream changes the shape.

You don’t own the source. One day a backend dev renames a column, and your pipeline either crashes, or worse, keeps running with wrong data.

Change upstreamWhat it does to youSeverity
Add a columnNothing, if you never SELECT *Safe
Drop / rename a columnRead of a missing column, hard failLoud
Change a column’s typeSilent cast / truncation, corrupt dataDangerous
Reorder columnsSELECT * misaligns every valueDangerous
Same shape, new valuesSchema check passes; aggregates silently wrongWorst
The dangerous ones are silent. A loud failure wakes you up; a quiet one ships wrong numbers for a month. The worst doesn’t touch the schema at all: a status grows a new value, an amount goes negative. That’s why shape-checking alone isn’t enough.
SCHEMA EVOLUTIONadd · drop · retype · reorder
Fix 04

Make drift loud. On purpose.

  • Never SELECT *. List every column by name, reordering and surprise columns can’t hurt you.
  • Pin a contract. Declare the columns + types you expect; check the live source against it before loading.
  • Fail fast. Missing/retyped column → stop with a clear error. New column → warn, keep going.
  • Then check values, not just shape. The contract catches a renamed column; it won’t catch a negative amount or an unknown status. Assert the invariants your aggregates assume: that’s data_quality.py.
schema_validate.py
for col, want_type in CONTRACT.items(): if col not in live: fail(f"missing column '{col}' (renamed or dropped upstream)") elif live[col] != want_type: fail(f"type drift on '{col}': {live[col]} != {want_type}")
opencode generate this column → type contract from information_schema for `orders`
SCHEMA CONTRACTexplicit columns · validate · fail fast

From script to system

A correct script isn’t a pipeline.

Say every problem above is solved. You still have one .py file. Now run it for 365 days, in the right order, with retries, and know when it breaks.

  • Scheduling: run at 02:00 daily, not cron on someone’s laptop
  • Dependencies: load only after extract and transform succeed
  • Retries + alerts: attempt again, then page a human
  • Backfill: re-run Jan 1–90 cleanly (needs idempotency!)
  • Observability: what ran, when, how long, why it failed
  • Lineage: which inputs produced which outputs
That bundle of concerns has a name: an orchestrator. We’ll meet two, Airflow and Dagster, next.
ORCHESTRATIONschedule · depend · retry · backfill · observe
The part everyone gets wrong

What an orchestrator won’t do.

It gives you
  • Scheduling & triggers
  • Task dependencies (the DAG)
  • Retries, alerting, backfill
  • A UI, logs, lineage
It does NOT give you
  • Idempotency
  • Correctness on re-run
  • Transactional loads
  • Schema safety
Retries are a trap without idempotency. A scheduler that re-runs a non-idempotent job just duplicates your data faster. The four fixes you just built by hand are the prerequisite, not the afterthought.

Orchestrator 1 · Airflow 3.2

Tasks and edges.

dags/daily_revenue_pipeline.py
default_args = {"retries": 3, "retry_exponential_backoff": True} with DAG("daily_revenue_pipeline", schedule="@daily", catchup=False, max_active_runs=1) as dag: # 1 = single-writer e = PythonOperator(task_id="extract", python_callable=extract) t = PythonOperator(task_id="transform", python_callable=transform) l = PythonOperator(task_id="load", python_callable=load) e >> t >> l
Map it back: retries=3 ↔ your retry loop · ds / dag_run.conf ↔ your target_date · >> ↔ calling the three in order · the metadata DB ↔ your watermark table. UI at :8080 (dev: no login).

The punchline

The framework didn’t make it correct.

con.execute("BEGIN TRANSACTION") con.execute("DELETE FROM daily_revenue WHERE date = ?", [ds]) for r in agg: con.execute("INSERT INTO daily_revenue VALUES (?, ?, ?, ?, ?, ?)", r) con.execute("COMMIT")

Take-home

Ship a pipeline you can trust.

  • Build the load right: DELETE + INSERT (or UPSERT) in one transaction, a watermark, SCD2 for one dimension, a schema contract, and a value-level data-quality check.
  • Prove it: prove_idempotent.py must print an identical checksum on every run, that’s the bar, not the prose.
  • Wrap it: Airflow or Dagster, your pick, and map each framework concept back to your hand-written code.
  • Submit: repo + README + AGENTS.md/CLAUDE.md, via PR.
If you can crash it mid-run, re-run it three times, and the checksum never moves, you’ve built the thing this lesson is about.

Annex · gotchas

Read before you debug.

  • DuckDB is single-writer. Runner, Airflow, Dagster share one file, run one writer at a time (that’s why max_active_runs=1).
  • Airflow catchup=False on purpose, so it won’t backfill two years on a laptop. Use airflow backfill create --from-date --to-date for a bounded range.
  • Reset levels: delete data/analytics.duckdb to drop the target; docker compose down -v wipes the seed + Airflow metadata too.
  • Timezone bites the date filter. created_at::date uses the session zone, so a day’s row count shifts under UTC vs local. experiment_timezone.py shows it; pin a zone for reproducible backfills.
  • First docker compose up -d is slow, it builds the images and pulls Postgres ×2.
Same Postgres, wal_level=logical already on, so Lesson 5’s CDC demo runs against this exact source, no re-provisioning.

Annex · scripts (1/2)

Infrastructure & pipeline phases.

config.py: Shared connection config and schema setup. Defines Postgres DSN, DuckDB path, creates target tables. All scripts import from here.
seed_data.py: Generates ~1M orders and ~50k customers in Postgres using DuckDB's postgres extension for fast server-side COPY. Configurable order count, customer count, and date window.
probe_data.py: Shows current state of both databases: row counts, date ranges, sample data. Use --reset to drop DuckDB and re-seed Postgres.
pipeline_naive.py: Phase 1: extract → transform → blind append load. No idempotency, re-runs duplicate rows. Shows the problem.

pipeline_failure.py: Phase 3: injects random crashes mid-load to demonstrate transactional recovery. DELETE+INSERT in one transaction means ROLLBACK leaves the target unchanged.

Testing, validation, & edge cases.

prove_idempotent.py: Runs a pipeline N times and compares row counts + checksums. Proves idempotency (all runs identical) or detects duplication (naive loader).
scd2.py: SCD Type 2 merge for customer dimension. Detects changed customers, expires old versions, inserts new ones with valid_from/valid_to windows. Includes --simulate-move to mutate source and --show to view version history.
schema_validate.py: Declares expected source schema (column names + types). Queries information_schema.columns and diffs against the contract. Detects missing columns, type changes, and new columns.
data_quality.py: Value-level data quality checks on source and target (negative amounts, nulls, revenue mismatches). Catches issues schema contracts miss.
experiment_upsert_gap.py: Demonstrates the UPSERT orphaned key problem: if a key disappears from the source, UPSERT leaves the old row behind while DELETE+INSERT removes it.
experiment_timezone.py: Shows how created_at::date partition boundaries shift based on session timezone. Counts rows for a date across UTC, America/Los_Angeles, and Asia/Tokyo.
All scripts take --help. The pipeline phases build on each other: naive → idempotent → failure → watermark. Run them in order to see the fixes accumulate.
