Hour 1 · the central problem

A count is not like a map.

A stateless map takes an event, returns a row, and forgets. A count must remember. If the processor crashes after counting 1000 events but before writing, what happens on restart?

two restart scenarios, one is wrong
replay from event 1: count = 1000 correct, slow replay from event 950: count = 50 wrong # the crash left the sink at 1000 but the checkpoint at 950 # or the checkpoint at 1000 but the sink at 950
State makes failure recovery non-trivial. You now have two things to keep in sync: position in the input stream and accumulated computation. If they diverge, the answer is wrong.
OFFSET + STATEmust advance together
Hour 1 · the universal shape

Source → operator → state → sink.

  • Stateless operators (map, filter): still no checkpoint state; replay anywhere gives the same output.
  • Stateful operators (aggregations, joins, sessions): must snapshot state and offsets together so a restart resumes from a consistent point.
  • The checkpoint is the single source of truth that ties offset to state version.
TOPOLOGYstate is the memory of the operator

Hour 1 · the practical join

A stream of events, a Kafka topic as a table.

transactions (stream) ──┐ ├── join on customer_id ──> enriched_transactions customers (compacted) ──┘ compacted topic = Kafka keeps only the latest value per key = a mutable table that can be rebuilt from the log
The recommended path is a stream-static join: load the compacted customer topic as a snapshot when the Spark app starts. Customer updates after start are ignored until the app is restarted. This is acceptable for slowly-changing lookup data.

Hour 1 · where state lives

State has to live somewhere.

EngineDefaultLarge state
FlinkJVM heapRocksDB (LSM-tree, spills to disk)
Spark StreamingHDFS/S3-compatible state storelocal disk in standalone mode; HDFS/S3 in cluster
On each micro-batch Spark reads the previous state version, applies the batch, writes a new state version, and commits the offset. The new state version is only as durable as the store it’s written to; in our lab it is a local directory, in production it is HDFS or S3.

Hour 1 · fault tolerance

Checkpoint = snapshot of offset + state.

one micro-batch
1. read previous state (from checkpoint store) 2. process new micro-batch 3. write updated state as a new version 4. commit offset to checkpoint 5. write output to sink # crash between 4 and 5? on restart replay the same batch # same state version + same offset = deterministic result
Exactly-once processing inside Spark: replayable source (Kafka) + checkpointed state + deterministic processing = after recovery, the internal state is identical to what it would have been without the failure.
CHECKPOINToffset + state version
Hour 1 · the critical gap

Exactly-once processing ≠ exactly-once delivery.

Kafka ──[replayable]──> Spark ──[at-least-once]──> Postgres (exactly-once (checkpointed (sink can receive processing) state) duplicates)
Spark can crash after writing to Postgres but before committing the checkpoint. On restart it replays the batch and writes the same rows again. The Postgres output now has duplicates. Spark’s guarantee stops at the sink boundary.
THE GAPprocessing stops at the sink
The impossibility
Two independent systems cannot atomically agree.

The two generals problem proves that no finite protocol can guarantee that Spark and Postgres commit the same outcome over an unreliable channel. True end-to-end exactly-once is impossible in the general case.

The practical solution is not to solve the impossible problem. It is to make the consequence of the duplicate disappear: write the same row twice, and the result is the same as writing it once. That is the idempotent sink.
TWO GENERALSno atomic commit across systems
Hour 1 · the practical answer

Make the duplicate harmless.

If the sink can accept the same row twice and end up with the same answer, the pipeline is effectively exactly-once. The implementation is a primary-key upsert.

the idempotent Postgres sink
INSERT INTO enriched_transactions (transaction_id, customer_id, amount, ...) VALUES (...) ON CONFLICT (transaction_id) DO UPDATE SET amount = EXCLUDED.amount, customer_id = EXCLUDED.customer_id, ...
The same transaction written twice produces the same row. The database returns success both times, but the answer is unchanged. The crash is no longer a correctness problem.
IDEMPOTENT SINKON CONFLICT · same row twice = once
Hour 1 · the chain

End-to-end is only as strong as the weakest link.

Kafka (replayable) ---> Spark (checkpointed state) ---> Postgres (idempotent upsert) exactly-once processing at-least-once write + deterministic replay + ON CONFLICT = safe
The full chain is effectively exactly-once: the source can replay, the processor can resume from the checkpoint, and the sink can absorb duplicates. This is how real pipelines survive crashes.

Phase 4 · the sink

The sink is where the promise breaks or holds.

write_to_postgres (foreachBatch)
def write_to_postgres(batch_df, batch_id): rows = batch_df.collect() with psycopg.connect(POSTGRES_URL) as conn: with conn.cursor() as cur: for row in rows: cur.execute(upsert_sql, row) conn.commit() # upsert_sql uses ON CONFLICT (transaction_id) DO UPDATE
The transaction_id is the primary key. The same transaction can be written twice; the result is the same. The sink is idempotent. That is the lesson.

Part three

Kill it.

The pass/fail test. Hard kill the Spark driver while the transaction generator keeps running. Restart the job with the same checkpoint. Prove the count is still right and the duplicates are still zero.

KILL AND RESTARTkill -9 · restart · verify
Break 01 · the protocol

Run it, kill it, restart it, prove it.

kill-and-restart protocol
1. run verify.py and record the count (pre-kill) 2. find Spark driver PID: ps aux | grep lesson8-stateful-join 3. kill -9 <pid> 4. keep transaction generator running for 60s 5. restart: uv run python src/streaming_join.py 6. wait for catch-up, run verify.py again
Use kill -9, not kill -15. A SIGTERM lets the Spark driver finish the current micro-batch cleanly, hiding the at-least-once behavior that the test is designed to expose.
PROTOCOLkill -9 · same checkpoint

Break 02 · remove the upsert

Remove ON CONFLICT and run the test again.

plain INSERT, then kill -9
INSERT INTO enriched_transactions (transaction_id, ...) VALUES (...) -- no ON CONFLICT after kill and restart: Duplicates: 17 Postgres count: 1,564 > produced 1,547 # impossible without upsert
The duplicates correspond to the partially written micro-batch at the moment of the kill. This is exactly the at-least-once behavior that the idempotent upsert hides. The difference between a demo and production is one line of SQL.

Synthesis

The three requirements.

RequirementProvided byHow it fails
replayable sourceKafka log + consumer offsetstopic retention too short
checkpointsSpark state store + offsetsshared checkpoint dir, or deleted
idempotent sinkPostgres ON CONFLICTmissing primary key, no upsert
Break any one of the three and the kill-and-restart test fails. The idempotent sink is the most commonly forgotten, because the engine and the source are so good that they hide the problem until the sink is overloaded or the process dies mid-batch.
SYNTHESISsource · checkpoint · sink
Synthesis · the alternative

The only way to get real exactly-once: 2PC.

two-phase commit
1. prepare: all participants vote "yes, I can commit" 2. commit: all participants must commit 3. if any participant is uncertain, the coordinator blocks # slow, fragile, coordination-heavy — and still has a failure window
Two-phase commit is the theoretical answer, but it is slow and operationally fragile. Most production systems prefer idempotent sinks: simpler, faster, and good enough because the result is the same.
2PCtrue exactly-once, practically avoided
What's next

From micro-batch to true streaming.

Spark Structured Streaming is a micro-batch engine. The checkpoint and state model are strong, but latency is bounded by the batch interval. Lesson 9 moves to Flink: record-at-a-time processing, event-time watermarks, and the same exactly-once guarantees with lower latency.

The hard part does not go away. State, checkpoints, and idempotent sinks are still the answer. The engine just changes the batch size from seconds to events.
