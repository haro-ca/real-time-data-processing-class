Phase 0 · the obvious idea

“Just sync what changed.”

Don’t re-read a million rows every night, only pull the ones touched since last time. Add an updated_at column, remember a high-water mark, loop. This is polling CDC, and it’s where everyone starts.

Phase 0 · the diagnosis

It’s the wrong question.

The drift isn’t three bugs to patch, it’s one impossible ask. “What changed since T?” is a question the table physically cannot answer honestly:

  • Deletes leave no trace. A removed row isn’t in any SELECT. You’d need soft-deletes everywhere, your OLTP schema now warped by sync needs.
  • Commit order ≠ updated_at order. A txn that started at T=100 but commits at T=105 is invisible to a poll at T=103. You miss it, or you bolt on fragile high-watermark margins.
  • Every poll competes with production. Poll often and you tax the OLTP box; poll rarely and you’re stale. There is no good point on that dial.
Patching each symptom just moves the lie around. The fix isn’t a better query, it’s a different direction of information flow.
DIAGNOSISpull can't answer honestly

Polling is pull: you interrogate the table on a timer. Log-based CDC is push: the database hands you a continuous, ordered stream of every committed change, deletes included, exactly once each.

Where the stream comes from

You’ve had the stream since Lesson 1.

Remember the WAL? Every committed transaction writes its changes to the write-ahead log before Postgres acknowledges the client. We met it as a durability mechanism. It’s also something else:

The WAL is already a complete, ordered log of every change to every row, an event stream hiding in plain sight.

CDC doesn’t add a new mechanism to Postgres. It reads the one that was always there. The only question is how to lift raw WAL back up to “row 42 went pending → shipped.”
THE WALdurability log = change stream
wal_level

Three levels. Only one decodes.

wal_levelWhat it recordsGood for
minimalJust enough to recover after a crashSingle node, no replication
replicaPage-level bytes: “write these bytes at this offset”Physical replicas, byte-for-byte
logicalTable, op, column values, before & afterCDC, this is what we need
A replica-level WAL says “write these bytes to that page”, it’s below the level of tables and rows, so you can’t reconstruct “status: pending→shipped” from it. Logical decoding lifts the WAL back up to table + operation + values.
WAL_LEVELlogical = decodable changes

  • Replication slot — a named, durable cursor into the WAL. Postgres won’t recycle WAL the slot hasn’t consumed, so you can disconnect and resume with zero loss. (Also: a forgotten slot fills the disk. More later.)

A setting that bites

Do you need the old values?

By default, an UPDATE/DELETE event carries only the primary key in its “before” image. You learn which row changed and its new values, but not what it used to be.

psql
-- include all old column values in the before-image ALTER TABLE orders REPLICA IDENTITY FULL;
  • Maintaining a materialized view? New values are enough, the default is fine.
  • Auditing or event sourcing? You need before & after, set REPLICA IDENTITY FULL, and accept fatter WAL entries.

Phase 2 · the sink

You already built the apply logic.

Map each event onto DuckDB. For updates, use DELETE + INSERT, the exact idempotent move from Lesson 4. Apply the same event twice and the row is identical.

cdc_consumer.py, DuckDBSink
def apply(self, ev): if ev["action"] in ("I", "U", "D"): duck.execute("DELETE FROM orders WHERE id = ?", [ev.id]) if ev["action"] in ("I", "U"): duck.execute("INSERT INTO orders VALUES (?, ?, ?, ?, ?)", ev.row) # delete runs for I too: a replayed INSERT is a no-op, not a dup key.
Why not plain UPDATE ... SET? Because any CDC consumer can crash and replay events. Idempotent apply means replay is harmless, the property that lets you sleep, again.
THE SINKdelete+insert = idempotent
Phase 3 · the main loop

The watermark, handed to you.

In Lesson 4 you computed a watermark to know what was already loaded. Here Postgres keeps it for you: confirmed_flush_lsn. The loop is three verbs: peek at pending changes (advances nothing), apply them, and only then advance the slot.

cdc_consumer.py, the loop
rows = pg.execute("SELECT lsn, data FROM pg_logical_slot_peek_changes( 'orders_slot', NULL, 5000, ...)") # 1. peek — advances nothing for lsn, data in rows: sink.apply(json.loads(data), lsn) # 2. write it pg.execute("SELECT pg_replication_slot_advance( 'orders_slot', %s)", [last_commit_lsn]) # 3. THEN confirm
Advancing tells Postgres “recycle WAL up to here.” Advance before applying and a crash loses the change; never advance and the slot retains WAL forever. And advance only to a commit marker’s LSN, mid-transaction positions aren’t safe points. Rule: apply, then confirm. Kafka makes you play this exact game again; there it’s called committing your offset (Lesson 6).

Break 01 · the missing million

The slot only sees the future.

Your consumer is perfect, yet DuckDB has 4 new rows and Postgres has 1,000,004. The slot started at its consistent_point; the million rows that existed before it never flowed through.

the gap
postgres orders 1,000,004 duckdb orders 4 ← only post-slot changes
A change stream is not a copy. To build a full mirror you need a one-time snapshot of what was already there, stitched to the stream with no gap and no overlap.
SNAPSHOT GAPstream ≠ copy
Break 01 · the fix

Snapshot the past. Replay seals the seam.

Create the slot first, so the stream is already buffering everything that happens next. Then bulk-copy the table, one SELECT, one MVCC snapshot. The seam: a few changes land in both the copy and the stream. We don’t fight that, the idempotent sink replays them into a no-op.

snapshot.py
# 1. slot already exists → every change since is buffered for replay duck.execute("ATTACH 'postgres://...' AS pg (TYPE postgres)") duck.execute("INSERT INTO orders SELECT ... FROM pg.orders") # the past # 2. drain the slot: overlapping events re-apply as delete+insert → no-op $ uv run python src/cdc_consumer.py # the future
Debezium’s “initial snapshot” phase gets an exact, no-overlap stitch by exporting the slot’s snapshot over the replication protocol. We trade that ceremony for plain SQL plus idempotency, and converge to the same mirror. Either way the snapshot can run for hours on a big table; now you know what it’s doing and why.
SNAPSHOT, THEN DRAINidempotency absorbs the seam
Break 02 · live

Kill it mid-stream. Restart. Same state.

Ctrl-C the consumer while the generator runs. Wait. Restart. It resumes from the last confirmed LSN, replays a few events, and the checksum holds.

$ uv run python src/cdc_consumer.py # after a kill
applied 50,123 events ... ^C (killed) restart -> resuming from confirmed LSN 0/3A2F108 replayed 12 events already in DuckDB (no-op, idempotent) checksum b04a42b3c882 == pre-crash b04a42b3c882 AT-LEAST-ONCE: no loss, replay harmless.
CDC is at-least-once, the default almost everywhere. You don’t lose events; you may re-see a few. The idempotent sink from Part one is what makes that safe instead of corrupting. Getting to exactly-once end-to-end, when you can’t lean on an idempotent sink, is the entire subject of Lesson 8.

Break 03 · the moving shape

A column appears mid-stream.

Upstream runs ALTER TABLE orders ADD COLUMN notes TEXT. The stream doesn’t break, wal2json simply starts emitting the new field. Your consumer has to notice and react.

  • Detect the unseen key in the event payload, compare against the columns DuckDB knows.
  • Adapt: issue ALTER TABLE ... ADD COLUMN on the target, the production choice, and keep going.
  • Or crash loud, if silent divergence is worse than an outage for your use case.
CDC is a schema contract between producer and consumer. Evolution is the hardest operational problem in streaming, and the reason schema registries exist. We meet them properly in Lesson 11.

Failing loud costs nothing here, and that’s not luck: the slot only advances after apply, so the crash leaves the event waiting. Confirm-after-apply is what makes “stop” a safe answer to drift.

Break 04 · live

A forgotten slot is a time bomb.

Create a slot, never consume it, run the generator. Postgres can’t recycle WAL the slot hasn’t confirmed, so it piles up until the disk fills and writes stop.

$ uv run python src/experiment_abandon_slot.py
slot 'abandoned_slot' created, nothing is reading it retained_wal 120 MB ... 480 MB ... 2.1 GB postgres: FATAL: could not write to WAL, no space left fix -> SELECT pg_drop_replication_slot('abandoned_slot');
An unmonitored slot can take down the source database. max_slot_wal_keep_size (PG 13+) is the safety valve: it drops a greedy slot to save the DB, at the cost of a forced re-snapshot.
DISK-FULLslot retains WAL → outage
Break · the one rule

Every slot needs a monitored consumer.

A replication slot is a promise the source keeps WAL for you. Break the promise, by abandoning or stalling the consumer, and the source pays in disk. So the operational contract is short:

  • Alert when confirmed_flush_lsn falls behind pg_current_wal_lsn().
  • Set max_slot_wal_keep_size so a stuck consumer can never sink the database.
  • Keep the apply path idempotent, so the inevitable replay is a no-op.
THE RULEno slot without a watcher

Looking ahead · the outbox

One more trick CDC unlocks.

A classic distributed-systems trap: an app must both write to the DB and publish an event. Do them separately and one can succeed while the other fails. The outbox pattern dissolves it.

one local transaction, no 2PC
BEGIN; INSERT INTO orders (...) VALUES (...); INSERT INTO outbox (event_type, payload) VALUES (...); COMMIT; # CDC streams the outbox row downstream, exactly-once
A distributed-consistency problem becomes a local transaction plus a CDC read. This is the backbone of Lessons 6–7, once Kafka is the thing reading the stream.

Read before you debug.

  • wal2json must be present. Vanilla postgres:16 ships pgoutput + test_decoding only; our image bundles wal2json. Swap to test_decoding if you can’t install it.
  • wal_level=logical needs a restart and bumped max_replication_slots / max_wal_senders. Already set in Compose.
  • One slot, one reader. Two consumers on the same slot will fight; give each its own slot (and double the WAL cost, the L6 motivation).
  • Two ways to read a slot. Plain SQL, peek_changes + slot_advance, is what we use. The streaming replication protocol (replication=True, send_feedback) is what Debezium uses, same confirm-after-apply contract, more plumbing.
  • Drop slots you stop using. An idle slot keeps retaining WAL until it fills the disk.
