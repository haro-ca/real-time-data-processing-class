04 / 30
CAP Consistency, Availability, Partition tolerance. During a partition you must choose C or A.
Theory

CAP theorem, done properly

  • During normal operation, you can have both C and A
  • During a network partition, you must choose one
  • Partitions will happen: cables, switches, GC pauses
  • The real question: refuse requests (CP) or serve stale data (AP)?
CP — CockroachDB
  • Minority side stops accepting writes
  • No stale reads, no split-brain
AP — Cassandra
  • Both sides accept writes
  • Reconcile later (last-write-wins)
05 / 30
The OLTP answer

For bank balances, AP with last-write-wins
can literally lose money.

Two nodes independently accepting transfers on the same account = inconsistent state. CP refuses the write rather than risk it.

Kleppmann’s critique: CAP is about a very specific consistency (linearizability) and a very specific availability (every non-failing node responds). Real systems operate between these extremes.
06 / 30
Quorum The minimum number of nodes that must acknowledge a write for it to be considered committed. For N nodes: ⌊N/2⌋+1.
Consensus

How do nodes agree?

  • Leader election: term numbers, vote requests, split-brain prevention
  • Log replication: leader appends, sends AppendEntries RPCs, waits for majority ACK
  • Safety: new leader always has all committed entries
  • 3 nodes → quorum = 2. 5 nodes → quorum = 3. Latency scales slowly.
Every write = leader + 1 network round-trip to a follower (minimum)
07 / 30
Raft · the happy path

Leader + 1 fast follower = committed. The slow follower catches up later.

08 / 30
Consensus · failure

What if the leader dies?

  • Detection: followers timeout on heartbeats, start election
  • Election: follower with the most complete log wins
  • Recovery: new leader replays uncommitted entries to lagging nodes
  • Old leader returns? Truncates its log to match the new leader
✓ No committed data is lost. The safety property guarantees this. You’ll verify it yourself in the kill-node demo.
⚠ But: election takes seconds. During that window, writes to affected ranges stall. This is the latency spike you’ll observe.

Single-key writes → Raft

What about transactions that span multiple nodes?

Transfer $100 from account A (node 1) to account B (node 2).
Both must succeed or neither. Enter: Two-Phase Commit.

10 / 30
2PC Two-Phase Commit — PREPARE all participants, then COMMIT or ABORT atomically.
Distributed transactions

Two-Phase Commit

Two network round-trips on the critical path. Each one is ~500 μs in the same datacenter.

11 / 30
Blocking A protocol is blocking if a single node failure can leave others unable to proceed — they must wait indefinitely for recovery.
The problem

2PC is blocking

  • Coordinator sends PREPARE, participants reply YES and hold locks
  • Coordinator dies before sending COMMIT
  • Participants are stuck: promised to commit, don’t know the decision
  • Locks held indefinitely. Known since the 1980s.
✓ CockroachDB solves this with parallel commits: the coordinator writes the transaction status to a distributed record in parallel with the final batch of writes, so participants can discover the outcome independently.
Modern distributed SQL hasn’t eliminated 2PC. It’s made the failure window vanishingly small and recoverable.
12 / 30
HLC Hybrid Logical Clock — combines physical time with logical counters for causal ordering.
Clocks

Clocks and ordering

In single Postgres: one incrementing XID counter. In a distributed system: who says txn 1000 on node A happened before txn 1001 on node B?

  • Lamport clocks: logical only, no wall-time relation. Sufficient for causal order.
  • HLC (CockroachDB): physical time + logical counter. Best-effort real time.
  • TrueTime (Spanner): GPS + atomic clocks. Bounds uncertainty to microseconds.
⚠ CockroachDB --max-offset = 500ms. If clocks drift beyond that, the node self-terminates. NTP misconfiguration has taken down production clusters.

Architecture

CockroachDB: ranges

  • Data split into ranges (default 512 MB)
  • Each range = one Raft group, replicated across 3 nodes
  • One node holds the leaseholder (like a Raft leader) per range
  • As data grows, ranges split and rebalance automatically
Node 1 Node 2 Node 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Range A* │ │ Range A │ │ Range A │ │ Range B │ │ Range B* │ │ Range B │ │ Range C │ │ Range C │ │ Range C* │ └──────────┘ └──────────┘ └──────────┘ * = leaseholder (leases spread across nodes)

Observation

Zero data loss

  • TPS dip: brief spike during leader re-election (seconds)
  • Recovery: surviving nodes pick up the load automatically
  • Data integrity: SUM(balance) before = SUM(balance) after
  • Node return: Raft log replay catches it up, no manual intervention
Kill 1 of 3
  • Quorum maintained (2/3)
  • Writes continue
  • Automatic failover
Kill 2 of 3
  • Quorum lost (1/3)
  • Writes refused (CP guarantee)
  • Consistency preserved

23 / 30
SSI Serializable Snapshot Isolation — CockroachDB’s default. Detects conflicts, aborts one side.
Retries

Serializable isolation = application retries

  • CockroachDB uses serializable by default (not read-committed)
  • Conflicting transactions get aborted, not queued
  • The application must catch 40001 errors and retry the whole transaction
  • Postgres queues behind locks. CockroachDB says “try again.”
retry pattern
for attempt in range(5): try: async with conn.transaction(): await do_transfer(conn, a, b) break except SerializationError: continue # retry

The answer

When to distribute?

  • Fault tolerance across failure domains and willing to pay the latency premium
  • Data volume exceeds what a single node can store
  • Multi-region writes are a hard requirement
⚠ If you just need more TPS: read replicas, connection pooling, or application-level sharding first. A single Postgres on good hardware often beats a 3-node CockroachDB on raw throughput.
✓ Lesson 1 showed the ceiling. Lesson 2 showed the cost of the alternative. That tension drives the rest of the course.

Serializable vs. Read Committed

CockroachDB (Serializable)
  • Conflicts detected at commit time
  • Losing transaction is aborted
  • App must retry (error code 40001)
  • Strongest guarantee: no anomalies
Postgres default (Read Committed)
  • Conflicts resolved by lock queue
  • Transactions wait, not abort
  • No app-side retry logic needed
  • Weaker: phantom reads possible
⚠ Different isolation levels = different application code. Migrating from Postgres to CockroachDB isn’t just a DSN change if you have any contention.
