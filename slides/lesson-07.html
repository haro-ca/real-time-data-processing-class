THE BRIDGEtransport → compute
Hour 1 · the universal shape

Source → operators → sink.

Kafka topic ──▶ parse ──▶ filter ──▶ window + aggregate ──▶ console / Kafka / DB (source) └────── stateless ──────┘ └─ stateful ─┘ (sink) no memory between events remembers across events
  • Stateless operators (map, filter, flatMap): output depends only on the current event. Trivially parallel, trivially restartable. The easy 20 minutes of today.
  • Stateful operators (aggregations, joins, sessions): output depends on what came before. State must survive crashes — that machinery is Lesson 8. Today we touch its simplest case: the windowed aggregate.
  • The topology is the same in Spark, Flink, and Kafka Streams. APIs differ; this mental model transfers.
TOPOLOGYa DAG over an endless input
Hour 1 · the tool

Spark, on purpose.

  • You will meet it at work. Structured Streaming is the most deployed stream processor in data engineering jobs, full stop.
  • You already think in DataFrames (pandas, DuckDB). The conceptual leap is one sentence: batch processing, but the input never ends.
  • Micro-batch makes time visible. Each batch is a discrete step you can watch — windows close, watermarks advance, between batches. A continuous engine hides this.
Micro-batch is also a latency floor — and whether that floor matters is a measurable question, not an opinion. We measure it in Lesson 9, Spark vs Flink, CDFs in hand. Today, micro-batch is the teaching instrument.
THE TOOLPySpark Structured Streaming
Hour 1 · the model

One word turns the table infinite.

A Spark DataFrame is a lazy, distributed table: transformations build a plan, nothing runs until something triggers it. The streaming version differs by four letters:

read vs readStream
spark.read.format("kafka")...load() # finite — a snapshot, then done spark.readStream.format("kafka")...load() # infinite — keeps ingesting # both yield the same columns: # key · value (binary!) · topic · partition · offset · timestamp
Almost every DataFrame operation works unchanged on the streaming one. The columns are Kafka’s — you recognize all six from last week, and value is raw bytes you must parse, exactly like your json.loads in the poll loop.
THE MODELlazy table, unbounded input
Hour 1 · the engine

An infinite loop of batch jobs.

the micro-batch loop, conceptually
every trigger interval ("5 seconds"): new_events = poll(kafka, since=last_offsets) # your L6 loop result = run_batch_query(new_events) # plan → optimize → execute write(result, sink) save(offsets + state, checkpoint_dir) # your commit, durably
Last week’s mapping table, honored. The poll loop, the offset commits, the resume-after-crash — all still happening, inside the trigger. The checkpoint directory plays the role of __consumer_offsets, plus it stores state. Internalize this loop and everything today — windows, watermarks, output modes — is just bookkeeping bolted onto it.
The trigger interval is a latency knob: shorter = fresher results, longer = bigger batches that amortize planning overhead. Hold that thought, we’ll prove it doesn’t change correctness.

Stateless: boring on purpose.

Identical to batch DataFrame code — that’s the point. No memory between events means nothing to checkpoint, nothing to lose, nothing to teach. The hard part starts when the answer depends on more than one event: aggregation. And aggregation over an endless table forces the question of time.
STATELESSselect · filter · explode
Hour 1 · the most important concept

When it happened vs when you saw it.

Event time

created_at — when the order actually happened, stamped by the source. The business truth.

Processing time

The wall clock when Spark sees the event — after network latency, consumer lag, CDC lag. An accident of your pipeline’s mood.

Window by processing time and your numbers describe when your pipeline felt like working, not when business happened. You proved the premise yourself in Lesson 6: Kafka preserves arrival order, and your shuffled topic arrived in the wrong order on purpose. That topic is today’s adversary.
EVENT TIMEthe business clock, not the pipeline’s
The concrete failure

Flash sale, 12:00–12:05. Pipeline lags 30 s.

Marketing wants revenue for the sale window. Your pipeline windows by processing time. Predict what the two windows report.

processing-time windows, 30s of lag
[12:00, 12:05) missing the last 30s of the sale — sale undercounted [12:05, 12:10) + 30s of sale revenue it never earned — post-sale inflated # same data, windowed by event time (created_at): both numbers correct
Not slightly wrong — structurally wrong, and it gets worse exactly when traffic spikes (lag grows during the moments you most care about). This is why every windowing expression today groups by created_at.

Hour 1 · windowing 1/3

Tumbling: every event, one window.

[12:00 ──── 12:05)[12:05 ──── 12:10)[12:10 ──── 12:15) ● ● ● ● ● ● ● ● non-overlapping, fixed size, gap-free — "revenue per 5-minute interval"
the aggregation
revenue = (orders .groupBy(window(col("created_at"), "5 minutes")) # event time! .agg(sum("amount").alias("total_revenue"))) # window() yields a struct column: {start, end}

The window is just a derived group-by key. The aggregation machinery is the GROUP BY you’ve used since Lesson 3 — what’s new is that the groups are open until time says otherwise.

SLIDING · SESSIONoverlap costs · gaps wait
The question every window asks
It’s 12:06. Is the [12:00, 12:05) window done?

Maybe — unless an event from 12:04 is still in transit. Emit now and you’re fast but possibly wrong. Wait and you’re right but late. Wait how long? Forever? Completeness versus latency — you cannot have both, so you need a mechanism that picks a point on the line.

THE TRADEOFFcomplete xor fast
Hour 1 · the mechanism

The watermark is a promise.

“No event older than T will arrive anymore.” Spark computes it from the data: the maximum event time seen so far, minus a lateness allowance you declare:

one line, heavy consequences
orders.withWatermark("created_at", "10 minutes") # watermark = max(created_at seen) − 10 min # seen 12:15 → watermark 12:05 → window [12:00,12:05) is COMPLETE: # emit its result · drop its state · and anything older that arrives now…
…is dropped. Silently. No error, no log line, no dead-letter queue by default. The watermark is also what lets Spark forget — without it, every window stays open and state grows until the process dies. The promise cuts both ways.
WATERMARKmax event time − allowed lateness

stream_revenue.py — the core

Live · output modes

Switch update to append. When does the first row print?

update emits partial results every batch — you watched windows fill up. Now restart in append mode, which only emits final results. Predict the delay.

append mode, 5-min windows, 10-min watermark
batch 0 … (empty) batch 1 … (empty) batch 2 … (empty) # nothing — until the watermark passes a window's END. # first output ≈ window(5m) + lateness(10m) of EVENT TIME later: batch 9: [12:00, 12:05) $14,382.91 427 orders final, immutable
That silence isn’t a bug — it’s the lateness tolerance working. append = clean, immutable rows for sinks that can’t update; update = upserts for dashboards and dev. (complete re-emits the whole table each batch; fine for tiny demos only.)

Live · the independence proof

Trigger 1 s → 30 s. Do the totals change?

Same pipeline, three runs: processingTime of 1, 5, and 30 seconds. Predict the effect on the window results before we run it.

three runs, same data
trigger=1s many tiny batches, results trickle in trigger=5s fewer batches, same numbers trigger=30s few big batches, same numbers [12:00, 12:05) = $14,382.91 in all three runs # identical. always.
The trigger controls latency. The window controls the question. They never trade. Confusing them produces pipelines with 1-second triggers “for accuracy” (waste) or 5-minute windows “for speed” (wrong answers). You now have the receipt that they’re orthogonal.

Break 01 · the alarm

The counter that tells on the watermark.

$ uv run python src/watch_progress.py (reads query.lastProgress)
batch 47 input 312 rows/s processed 298 rows/s state: numRowsTotal 1,204 # open windows being remembered numRowsUpdated 96 numRowsDroppedByWatermark 1 # 50 late orders, ONE window = 1 group # it counts dropped GROUPS, not events — the $49,999.50 hides in a hot key
Every block of this course ends at one operational number: TPS (L1), slot lag (L5), consumer lag (L6) — and now numRowsDroppedByWatermark. Zero means your lateness allowance is generous enough. Nonzero means you’re shipping wrong numbers, knowingly — a business decision, not a bug. One caveat that is the lesson: it counts dropped aggregation groups, not events — 50 late orders in one window read as +1. Treat it as a tripwire (zero vs nonzero); recover the lost dollars with the batch audit.

Synthesis · the dropped 0.1%

Streaming for speed. Batch for truth.

Your watermark drops 0.1% of events. Acceptable? For a dashboard, probably. For finance, no. But notice what “dropped” means: dropped from the stream’s answer. The events still exist — in Kafka, in Postgres.

So the production pattern: the stream serves the fast, approximately-complete answer; the L4 batch pipeline — idempotent, re-runnable, the one you already trust at 3 AM — reconciles the truth nightly. Lesson 4 wasn’t replaced by streaming. It became the audit. You’ll see this pairing again in Lessons 10–11.

Annex

Read before you debug.

  • JAVA_HOME. PySpark needs Java 11+. brew install openjdk@17 and export the path. Nearly every “Spark won’t start” is this.
  • First run is slow. spark.jars.packages downloads the Kafka connector from Maven once. Not a hang — wait it out, it caches.
  • Seeing nothing? Default startingOffsets is latest — new events only. Use earliest in the lab.
  • One checkpoint dir per query. Shared checkpoints corrupt both queries. Changed the aggregation logic? Delete the checkpoint, it pins the old plan.
  • The watermark is global. It follows the max event time across all partitions — one stalled partition won’t hold it back, and that partition’s events may be dropped when it catches up. Operational gotcha, rare in the lab, real in production.
  • Dev in update mode. append’s silence-until-watermark looks exactly like a broken pipeline while you’re iterating.
