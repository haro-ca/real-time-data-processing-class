The idea

A log in the middle.

Put a durable, replicated, append-only log between producers and consumers. Producers append and walk away. Each consumer keeps its own position and reads at its own pace. A slow reader inconveniences nobody, not the source, not the other readers.

It’s a different data structure, not a faster queue. Every property in this lesson, fan-out, replay, consumer lag, rebalancing, follows from “append-only log + consumer-owned offsets.” Get this and the rest is corollary.

Why it’s fast

Append-only is the physics answer.

  • Writes: append to the end of a segment file. No seeks, no rewrites. NVMe sustains 2–3 GB/s sequential.
  • Reads: consumers read forward from a position, also sequential. The OS page cache serves hot segments from RAM; Kafka manages no cache of its own.
  • No per-consumer state on the broker. It appends and serves ranges. All the bookkeeping a queue does per message simply doesn’t exist.
You measured this physics in Lesson 1: COPY’s sequential batches beat row-by-row INSERT by 50× on the same disk. Kafka is that lesson built into an architecture — align the data structure with what disks are good at, and millions of messages per second on modest hardware stops being magic.
STORAGEsequential I/O + page cache
The unit of everything

Topic, split into partitions.

A topic is a named log. One log on one machine doesn’t scale, so Kafka splits it into partitions: independent logs on (potentially) different brokers. The offset is just an integer, your position in one partition.

Topic: orders (3 partitions) P0: [0][1][2][3][4][5][6]→ key="cust-77" → hash % 3 = 0 P1: [0][1][2][3][4]→ key="cust-42" → hash % 3 = 1 P2: [0][1][2][3][4][5][6][7][8]→ key="cust-13" → hash % 3 = 2
  • Inside a partition, order is guaranteed. Offset 42 was written before 43. Always.
  • Across partitions, there is no order. None. That’s the price of parallelism.
  • Same key → same partition → per-key order. No key → sticky batches, near-uniform spread.
TOPICS · PARTITIONSorder inside, parallel across
The decision that bites

The key is an architectural decision.

  • Key by customer_id: all of a customer’s events stay ordered, but a whale customer concentrates load on one partition.
  • Key by order_id: per-order lifecycle stays ordered; a customer’s orders may interleave across partitions.
  • Need total order across everything? One partition. One broker’s throughput. That ceiling is a law, not a config knob.
You’ve met this twice: the hot row that collapsed Postgres in L1, and the shard key that decided everything in L3. A hot key makes a hot partition, same disease, new organ. Choose keys the way you chose shard keys: by how the load distributes, not by what’s convenient.
KEYShot key = hot partition
How reading scales

Consumer groups divide the partitions.

A consumer group is a team reading one topic together. Kafka gives each partition to exactly one consumer in the group. Different groups are invisible to each other, each gets the whole stream.

group "order-processor" 6 partitions 2 consumers: A → P0 P1 P2 B → P3 P4 P5 3 consumers: A → P0 P1 B → P2 P3 C → P4 P5 7 consumers: six work, one sits idle ← the hard ceiling
Parallelism = partition count. You choose it at topic creation, and changing it later reshuffles keys and triggers rebalances. This is the capacity-planning decision of Kafka, make it with headroom.

Offset commits

The same rule. Third time.

A consumer periodically commits “I’ve processed up to offset X” (to an internal topic, __consumer_offsets). On restart it resumes from there. You have built this watermark twice already, and the rule hasn’t changed:

LessonThe watermarkThe rule
L4 batchpipeline_metadata tablerecord “loaded” in the same txn as the load
L5 CDCconfirmed_flush_lsnapply, then advance the slot
L6 Kafkacommitted offsetprocess, then commit the offset
Commit before processing → crash loses messages (at-most-once). Commit after → crash replays a few (at-least-once, harmless if the sink is idempotent — your L5 move). The default enable.auto.commit=true commits on a 5-second timer, uncorrelated with your processing: it can do either. We turn it off all lesson.
OFFSETSprocess → then commit
The stop-the-world event

Rebalancing: partitions move.

A consumer joins, crashes, or just polls too slowly — and the group coordinator redistributes partitions. Two callbacks bracket the move, and one of them carries all the risk:

  • on_revoke — partitions are about to be taken. Your last chance to commit in-flight work. Skip it, and the next owner reprocesses everything since your last commit.
  • on_assign — new partitions arrive; initialize their state.
  • Cooperative-sticky (the modern strategy) moves only the partitions that must move; the rest keep flowing. The old eager protocol froze the whole group.
A botched on_revoke is the single most common source of duplicates in real Kafka systems — and it’s exactly what your take-home is graded on. We’ll trigger one live in Part two.
REBALANCEon_revoke = commit or duplicate

Replication

ISR: consensus with the dial exposed.

Each partition has a leader and followers. The ISR (in-sync replicas) is the set that’s caught up. With acks=all, a write is confirmed when every ISR member has it. The ISR shrinks when followers lag — and that’s the difference from Raft:

Raft (CockroachDB, L2)
  • Quorum = fixed majority, always
  • 2 of 3 nodes down → refuses writes
  • Consistency wins, no dial
ISR (Kafka)
  • ISR is a dynamic set, can shrink to 1
  • min.insync.replicas is the floor you choose
  • Floor=1: available but a dead leader loses data
the production durability triplet
replication.factor = 3 # three copies min.insync.replicas = 2 # refuse writes below leader+1 acks = all # producer waits for the ISR # lose 1 broker: fine · lose 2: writes refuse, on purpose
ISRavailability dial vs Raft’s fixed quorum

Phase 2 · consume_naive.py

The poll loop is the heartbeat.

consume_naive.py
consumer = Consumer({"bootstrap.servers": "localhost:19092", "group.id": "order-processor", "auto.offset.reset": "earliest", # first run only "enable.auto.commit": False}) # WE own the watermark consumer.subscribe(["orders"]) while True: msg = consumer.poll(timeout=1.0) # also the liveness signal if msg and not msg.error(): process(msg) consumer.commit(asynchronous=False) # process, THEN commit
If you don’t call poll() within max.poll.interval.ms (5 min default), the broker declares you dead and rebalances your partitions away — slow per-message work between polls is the classic self-inflicted rebalance. And committing every message is fine for learning, terrible for throughput; we batch it next.

Break 01 · consume_rebalance.py

Commit on the way out.

consume_rebalance.py
def on_revoke(consumer, partitions): # partitions are about to be taken — last chance for in-flight work consumer.commit(offsets=consumer.position(partitions), asynchronous=False) def on_assign(consumer, partitions): init_state(partitions) # fresh counters per partition consumer.subscribe(["orders"], on_assign=on_assign, on_revoke=on_revoke) # + "partition.assignment.strategy": "cooperative-sticky"
This is L5’s apply-then-confirm discipline meeting a new failure mode: the work doesn’t crash, it gets reassigned. The commit in on_revoke is the handoff note you leave for whoever inherits the partition.

Synthesis

One structure. Four rules.

Kafka is an append-only log with consumer-owned positions. Everything else you saw today is a corollary. The rules to carry out of the room:

  • Process, then commit. The watermark rule, in its third and final costume. Pair it with an idempotent sink and replay is free.
  • Keys are architecture. They decide ordering, balance, and who suffers when a customer goes viral. Decide before the first message.
  • Parallelism = partitions. Chosen at creation, expensive to change. Capacity-plan it like you planned shards.
  • Lag is the metric. Flat = healthy. Growing = scale consumers, or accept the delay. Nobody’s disk fills anymore, that’s the upgrade.

Annex

Read before you debug.

  • Advertised listeners. The broker you dial replies “talk to me at this address” — if that address isn’t reachable from your laptop, you connect, then mysteriously time out. That’s why each broker advertises its own port (19092/29092/39092). The #1 Docker-Kafka failure.
  • produce() is buffered. No poll()/flush() = no delivery, no error, no trace. Always flush before exit.
  • Stale volumes break cluster formation. Recreated the compose file? Old volume keeps the old cluster ID and brokers refuse to join. docker compose down -v.
  • Slow processing = rebalance storm. Exceed max.poll.interval.ms between polls and you’re evicted; the group rebalances, twice. Do heavy work in batches, poll often.
  • confluent-kafka, not kafka-python. The latter is unmaintained.
  • More consumers than partitions = idle consumers. Not an error, just silent waste. Check the assignment printout.
