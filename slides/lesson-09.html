Hour 1 · recap

Lessons 7–8 gave us correct results. Now: how late?

  • 5-minute tumbling windows over Kafka order events.
  • Watermark handles late events up to 5 seconds.
  • Output is correct, idempotent, and checkpointed.
  • Latency distribution is unknown — until we instrument it.
The question is not whether it works. It is whether it meets the SLA.
RECAPcorrectness ≠ latency
Hour 1 · Spark

Spark runs a tiny batch every trigger.

one micro-batch lifecycle
1. offset discovery # Kafka offsets per partition 2. micro-batch planning # build a Spark SQL DAG 3. task scheduling # serialize plan, launch tasks 4. data processing # pull records, compute windows 5. output commit # write sink, commit offsets 6. wait for next trigger
Latency floor = max(trigger_interval, batch_processing_time). A record arriving 1 ms after a batch starts waits for the next batch.

Hour 1 · Flink

Flink processes records one at a time.

  • Sources pull from Kafka partitions continuously.
  • Records flow through the operator DAG as they arrive.
  • Watermarks propagate; a window fires when the watermark passes its end.
  • Output is emitted immediately; checkpointing is asynchronous.
Latency floor = network + deserialize + operator processing + watermark delay. For a simple aggregation, often 10–100 ms.
FLINKtrue streaming
Hour 1 · proof

Proof: Flink's floor is the watermark.

Same read as the Spark slide: bars are individual closed windows, the dashed line is the configured watermark bound. Every bar sits just above its line, and the line itself moves as the watermark does (1s, 5s, 10s) — there is no trigger interval here to blame or tune instead, only the watermark you chose.

Hour 1 · gotcha

The gotcha: a quiet source stalls the watermark forever.

  • A bounded-out-of-orderness watermark only advances when new records arrive.
  • Burst of data, then silence, is a normal real-world pattern — end of day, sensor dropout, upstream hiccup.
  • Without with_idleness(): the watermark freezes at the last event's timestamp. Any window still waiting on it never fires. No error, no warning — it just never happens.
ConfigWindows closedTrailing window
with with_idleness(1s)2 / 2closed (max 5.3s)
without idleness handling1 / 2STALLED — never closed
Measured (src/demo_idle_source_stall.py): 2 windows of data, then 30s of silence. The second window sat open through the entire observation window without idleness handling. with_idleness() is not a tuning knob — treat it as required for any source that can go quiet.
GOTCHAidle source, watermark stall
Hour 1 · trade-off

Trigger interval vs. latency — what we measured.

Measured, not asserted: hold the rate fixed and sweep Spark's trigger (src/trigger_sweep.py). Spark's floor is max(trigger, batch_time), so its p50 climbs 7s → 8s → 15s → 20s as the trigger loosens; Flink has no trigger — its floor is the watermark, so it holds flat at ~7s. At a 1s trigger they nearly tie: micro-batch isn't inherently slower, it trades latency for the batching efficiency that wins at high throughput. Why sweep the trigger and not throughput? On one laptop the producer caps out (~586 events/s) before either engine saturates, so throughput never moved latency — the trigger is the knob that does.

Hour 2 · experiment

No wall. Spark's floor keeps falling.

  • Spark's p50: 20.0s → 15.0s → 8.0s → 7.0s → 6.0s → 5.0s as trigger falls 10s → 5s → 2s → 1s → 0.5s → 0.25s.
  • No plateau anywhere in range — the "diminishing returns below 2s" claim in an earlier draft of this deck was never measured, and turned out wrong.
  • At 250ms, Spark's floor (5.0s) lands almost exactly on the shared 5s watermark floor both engines are bounded by.
Don't over-read the Flink line. It bounces between 5.4s and 7.9s here, noisier than the tight ~5.6s demo_watermark_bound.py measured for Flink alone. Both engines share this machine's CPU in every round, and a tighter Spark trigger means more frequent planning/scheduling — that can perturb Flink's measured latency as a side effect, not a change in Flink's own architecture. Spark's downward trend is the reliable finding here; the exact crossover point against Flink is not.

Hour 3 · honesty

What this benchmark doesn’t prove.

  • PyFlink is slower than Java/Scala Flink.
  • Single-machine results hide real network shuffles.
  • State size matters: RocksDB spill changes everything.
  • Exactly-once overhead differs: Spark per-batch commit vs. Flink Chandy-Lamport checkpoints.
The shape of the curves holds: micro-batch has a latency floor; true streaming does not. The exact numbers will change at scale.

Takeaway

Choose by SLA, measure by CDF.

Micro-batch has a latency floor because of its batch boundary. True streaming does not. Neither is universally better; the right engine is the one that meets the SLA with the least operational cost.
