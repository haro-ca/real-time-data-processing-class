Hour 1 · recap

Lessons 7–8 gave us correct results. Now: how late?

  • 5-minute tumbling windows over Kafka order events.
  • Watermark handles late events up to 5 seconds.
  • Output is correct, idempotent, and checkpointed.
  • Latency distribution is unknown — until we instrument it.
The question is not whether it works. It is whether it meets the SLA.
RECAPcorrectness ≠ latency
Hour 1 · Spark

Spark runs a tiny batch every trigger.

one micro-batch lifecycle
1. offset discovery # Kafka offsets per partition 2. micro-batch planning # build a Spark SQL DAG 3. task scheduling # serialize plan, launch tasks 4. data processing # pull records, compute windows 5. output commit # write sink, commit offsets 6. wait for next trigger
Latency floor = max(trigger_interval, batch_processing_time). A record arriving 1 ms after a batch starts waits for the next batch.
SPARKmicro-batch overhead
Hour 1 · Flink

Flink processes records one at a time.

  • Sources pull from Kafka partitions continuously.
  • Records flow through the operator DAG as they arrive.
  • Watermarks propagate; a window fires when the watermark passes its end.
  • Output is emitted immediately; checkpointing is asynchronous.
Latency floor = network + deserialize + operator processing + watermark delay. For a simple aggregation, often 10–100 ms.
FLINKtrue streaming
Hour 1 · trade-off

Throughput vs. latency — what we measured.

Honest result: from 19 to 586 events/s on a single laptop, Spark stayed flat at exactly 8.0s and Flink stayed inside its own band — neither engine's latency responded to throughput in this range. The batching-amortizes-overhead trade-off is real in principle, but you will not see it until batch processing time actually exceeds the trigger interval — which took far more load, or a far tighter trigger, than this range. That is the next demo.

Hour 1 · code

The same window, expressed imperatively.

pyflink datastream
parsed.assign_timestamps_and_watermarks( WatermarkStrategy.for_bounded_out_of_orderness(Duration.of_seconds(5)) .with_timestamp_assigner(OrderTimestampAssigner()) ).key_by(lambda _: 'all') .window(TumblingEventTimeWindows.of(Time.minutes(5))) .process(WindowAggregation())
No DataFrame abstraction; explicit watermarks and a ProcessWindowFunction. The function records emit_ts_ms when the window fires — that is the latency measurement hook.

Hour 2 · analyze

Compute the CDFs.

generate report and plot
uv run python src/analyze_latency.py
  • Consumes results-spark and results-flink.
  • Computes p50 / p95 / p99 for both latency metrics.
  • Writes data/latency_cdf.png and data/latency_report.json.
Measured, not guessed (2s trigger, 15s window, 16/16 windows closed): Spark p50 8.0s / p99 9.0s. Flink p50 7.6s / p99 8.6s. Flink sits ~400-500ms lower across the whole distribution — not just the median. A flat, constant gap top-to-bottom would mean a measurement artifact; this gap tracks the shape of the curve.

The money slide

The CDF tells the story.

Measured on this machine: Flink’s curve sits left of Spark’s across almost the whole distribution — a consistently lower floor, not just a lower median.

Look at the whole curve, not just the median. Both curves step (quantization from the trigger/watermark clock), but Spark’s tail extends further right — and its floor never gets as low as Flink’s.

Hour 2 · experiment

Spark has a floor.

  • Run Spark with trigger = 10 s, 5 s, 2 s, 1 s, 500 ms.
  • Plot p99 latency versus throughput for each setting.
  • Diminishing returns appear below ~2 seconds.
As the trigger shrinks, per-batch overhead (offset discovery, planning, scheduling, commit) becomes a larger fraction of the work. Throughput can drop while latency barely improves.

Hour 3 · honesty

What this benchmark doesn’t prove.

  • PyFlink is slower than Java/Scala Flink.
  • Single-machine results hide real network shuffles.
  • State size matters: RocksDB spill changes everything.
  • Exactly-once overhead differs: Spark per-batch commit vs. Flink Chandy-Lamport checkpoints.
The shape of the curves holds: micro-batch has a latency floor; true streaming does not. The exact numbers will change at scale.

Takeaway

Choose by SLA, measure by CDF.

Micro-batch has a latency floor because of its batch boundary. True streaming does not. Neither is universally better; the right engine is the one that meets the SLA with the least operational cost.
