Module A

Connection overhead

  • One OS process per connection — not a thread
  • Each backend: ~5–10 MB RSS
  • 500 connections = gigabytes just in process overhead
  • Context switching kills throughput beyond the sweet spot
max_useful_concurrency ≈ (CPU cores × 2) + effective_spindle_count
⚠ This is why pgbouncer exists. Connection pooling isn't optional at scale.

Module B

WAL and the fsync bottleneck

  • Every commit requires a WAL flush to disk
  • Spinning disk: ~100–200 fsyncs/sec
  • NVMe SSD: ~100k+ IOPS
  • This is often the first bottleneck you'll hit
wal_level settings
  • minimal — least overhead
  • replica — streaming replication
  • logical — CDC (Lesson 5)
The cost
  • Each level writes more WAL detail
  • logical is most expensive
  • But it's what enables CDC
06 / 27
Module B

synchronous_commit = off

What you gain
  • Massive TPS improvement
  • WAL still written, just not fsynced per commit
  • Background writer flushes in batches
What you lose
  • Up to wal_writer_delay of committed txns on crash
  • Default: 200ms window of potential data loss
  • At 50k TPS = 10,000 transactions
💡 This is a real production tradeoff, not academic. You'll measure it in Experiment B.
07 / 27
Module C

Shared buffers & the double-buffering problem

  • Postgres manages its own buffer pool (shared_buffers)
  • OS also caches the same pages in the page cache
  • Too large shared_buffers = caching pages twice
  • Rule of thumb: 25% of RAM — because of this
⚠ For write-heavy workloads: if checkpoint I/O spikes, your write latency spikes. The checkpointer writes dirty pages from shared_buffers to disk.

Module D

MVCC: updates create new tuples

UPDATE orders SET amount = 100 WHERE id = 42; ┌─────────────────────────────┐ ┌─────────────────────────────┐ │ Old Tuple │ │ New Tuple │ │ xmin = 100 xmax = 105 │ │ xmin = 105 xmax = ∅ │ │ amount = 50 ← DEAD │ │ amount = 100 ← LIVE │ └─────────────────────────────┘ └─────────────────────────────┘
  • UPDATE-heavy workloads accumulate dead tuples
  • Table bloat grows — sequential scans get slower
  • This is invisible until you look for it
10 / 27
Module D

Vacuum: the cleanup cost

  • Vacuum reclaims dead tuple space
  • Does I/O, takes lightweight locks, competes with workload
  • Autovacuum defaults are conservative
  • Aggressive writes can outpace vacuum → bloat spiral
✓ You'll see this happen in Experiment D — disable autovacuum, watch bloat grow, then run VACUUM VERBOSE and see the reclaimed space.
11 / 27
Module E

Lock contention

  • Many transactions UPDATE the same hot row (inventory, balance)
  • Postgres uses a wait queue — serialized access
  • Beyond a threshold: more time waiting than working
  • No amount of hardware fixes this
⚠ Solutions are architectural: optimistic locking, application-level sharding of the hot row, queuing updates. You can't buy your way out of contention.

Theory

Amdahl's Law applied

Speedup = 1 / ( S + (1 − S) / N )

S = fraction of work that is inherently serial. N = parallel workers (connections).

  • WAL fsync is serial — one disk, one flush queue
  • Lock waits are serial — only one txn holds the row
  • Even with infinite connections, speedup is bounded by 1/S
  • If 5% of your workload is serial, max speedup = 20×
💡 This explains why adding connections past the sweet spot degrades performance — you're adding overhead without reducing the serial fraction.

Phase 1

Naive: one connection, synchronous

load_naive.py
import psycopg conn = psycopg.connect("dbname=bench") with conn.cursor() as cur: for i in range(100_000): cur.execute(""" INSERT INTO orders (customer_id, amount) VALUES (%s, %s)""", (i % 1000, 9.99)) conn.commit()
⚠ Expected: ~500–2,000 TPS. One connection, synchronous commits, no pipelining. Each INSERT round-trips and waits for WAL flush.

Phase 2

Async with N coroutines

load_async.py
import asyncio, asyncpg async def worker(pool): async with pool.acquire() as c: await c.execute(""" INSERT INTO orders (customer_id, amount) VALUES ($1, $2)""", 42, 9.99) async def main(n=50): pool = await asyncpg .create_pool(max_size=n) await asyncio.gather( *[worker(pool) for _ in range(100_000)] )
Expected TPS curve
💡 More connections ≠ more throughput.

Reference

How to read a flame graph

┌──────────────────────────────────────────────────────────────────────┐ │ asyncio.run() │ ├────────────────────────────────┬─────────────────────────────────────┤ │ event_loop.run_forever() │ │ ├───────────────┬────────────────┤ │ │ asyncpg │ _poll_socket │ gc.collect() │ │ .execute() │ (waiting I/O) │ (2% of time) │ ├───────┬───────┤ │ │ │ encode│ send │ │ │ └───────┴───────┴────────────────┴─────────────────────────────────────┘ ← wider = more time spent there →
  • X-axis = time fraction (wider = more samples = more time)
  • Y-axis = call stack depth (bottom = entry point)
  • Look for the widest leaf — that's your bottleneck

23 / 27

Experiments A & B

A · Batching
  • Use executemany or COPY
  • Fewer round-trips, fewer WAL flushes
  • One commit per batch, not per row
  • Expected: 5–10× TPS jump
B · synchronous_commit off
  • SET synchronous_commit = off;
  • TPS jumps significantly
  • What did you give up?
  • At 50k TPS × 200ms = 10k lost txns on crash
💡 Both reduce the serial fraction (WAL fsync). Batching is free. sync_commit off costs durability. Know the price.

Experiment C

The hot row collapse

SQL
-- Everyone hits the same row UPDATE orders SET amount = amount + 1 WHERE id = 1;
  • TPS collapses even at modest concurrency
  • Backends pile up in lock waits
  • More connections = worse performance
  • Hardware cannot fix contention
⚠ This is the most dramatic demo. Watch pg_stat_activity fill up with Lock wait events in real time.

Experiment D

Bloat: dead tuples accumulate

  • Run heavy UPDATE workload for a few minutes
  • Check pg_stat_user_tables.n_dead_tup
  • Disable autovacuum: ALTER TABLE orders SET (autovacuum_enabled = false)
  • Keep updating — watch table size grow
  • Sequential scan slows down (scanning dead rows)
✓ Then: VACUUM VERBOSE orders; — see the reclaimed space and recovered performance.
