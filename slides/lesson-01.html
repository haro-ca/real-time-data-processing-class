02 / 29
Autopsy

What just happened?

  • We opened 500 connections to a Postgres container
  • Each connection is a separate OS process (a “backend”)
  • Each backend allocated ~50 MB of private memory
  • Total: ~25 GB demanded. Container limit: 4 GB.
  • Linux OOM killer chose Postgres. Everything gone.
⚠ No graceful shutdown. No WAL (Write-Ahead Log) flush. In-flight work is lost. This happens in production.
03 / 29
RSS Resident Set Size — physical RAM a process actually occupies right now.
Investigation

What IS a connection?

  • Postmaster forks a new process for every client
  • Each backend: ~5–10 MB RSS (Resident Set Size) at idle, more under load
  • work_mem, temp buffers, sort buffers — all per-backend
  • 500 connections = gigabytes just in process overhead
  • Context switching kills throughput beyond the sweet spot
max_useful_concurrency ≈ (CPU cores × 2) + effective_spindle_count
💡 This is why pgbouncer exists. Connection pooling isn't optional at scale.

06 / 29
fsync System call that forces buffered writes to durable storage. Returns only when bits are on disk.
IOPS I/O Operations Per Second — the throughput limit of your storage device.
Bottleneck 1

The fsync wall

  • Every COMMIT requires a WAL flush to durable storage
  • fsync() = “don't return until bits are on the platter”
  • NVMe SSD: ~100k IOPS → ceiling of ~100k synchronous commits/sec
  • One connection, one INSERT at a time → one fsync per commit
⚠ This is typically the first bottleneck you'll hit. The disk becomes the serialization point.

08 / 29
Bottleneck 1 · workaround

Group commit: concurrency helps

  • Multiple backends committing → Postgres batches their WAL flushes
  • One fsync serves several commits → amortized cost
  • 10 connections: ~5× improvement. 50: ~8×.
  • Diminishing returns as contention on WAL lock grows

11 / 29
Bottleneck 1 · nuclear option

Removing the fsync brake

What you gain
  • COMMIT returns immediately
  • WAL still written, just not fsynced per commit
  • Background writer flushes in batches
What you lose
  • On crash: lose up to 200ms of committed data
  • At 50k TPS × 200ms = 10,000 txns gone
  • Not corruption — just acknowledged-but-lost commits
⚠ Don't believe it? demo_sync_loss.py inserts rows, crashes the container, and counts what's missing. Real proof.

13 / 29
COPY PostgreSQL’s bulk-load protocol. Streams rows in binary, bypassing per-row overhead.
Bottleneck 1 · eliminated

Batching: the COPY protocol

  • Instead of 1,000 round-trips → one bulk transfer
  • COPY streams rows in binary, one WAL flush per batch
  • Fewer fsyncs, fewer context switches, fewer network round-trips
  • executemany: pipelined INSERTs. Good. COPY: binary stream. Great.
✓ Batching is free — no durability trade-off. Just fewer operations per commit.

16 / 29
Theory

Amdahl's Law applied

Speedup = 1 / ( S + (1 − S) / N )

S = serial fraction. N = parallel workers. Plain English: any work that can only happen one-at-a-time sets a hard ceiling on total speedup — adding more connections can’t move that wall.

  • WAL fsync is serial — one disk, one flush queue
  • Lock waits are serial — only one txn holds the row
  • Even with infinite connections, speedup is bounded by 1/S
  • If 5% of your workload is serial, max speedup = 20×
💡 This explains why connections past the sweet spot degrade performance. More overhead, same serial fraction.
17 / 29
Bottleneck 2

The hot row collapse

  • Many transactions UPDATE the same row (inventory, balance)
  • Postgres uses a wait queue — serialized access
  • Beyond a threshold: more time waiting than working
  • No amount of hardware fixes contention
SQL
-- 50 backends, all hitting the same row UPDATE orders SET amount = amount + 1 WHERE id = 1;
⚠ Solutions are architectural: optimistic locking, sharding the hot row, queuing updates. You can't buy your way out.

Bottleneck 3

Shared buffers & the double-buffering problem

  • Postgres manages its own buffer pool (shared_buffers)
  • OS also caches the same pages in the page cache
  • Too large shared_buffers = caching pages twice
  • Rule of thumb: 25% of RAM — because of this
⚠ For write-heavy workloads: if checkpoint I/O spikes, your write latency spikes. The checkpointer writes dirty pages from shared_buffers to disk.

MVCC Multi-Version Concurrency Control — readers never block writers. Each row version has xmin/xmax transaction IDs.
Bottleneck 4

MVCC & dead tuples

  • UPDATE creates a new tuple. Old one becomes dead.
  • Dead tuples accumulate → table bloat → slower scans
  • Vacuum reclaims space, but competes with your workload
  • Autovacuum defaults are conservative — writes can outpace it

Instrument everything

Postgres side

  • iostat -x 1 — disk util & await
  • vmstat 1 — context switches, CPU
  • htop / ps aux — per-process view
  • dmesg — OOM kill evidence

Circle back

Why did it die?

  • Connections are expensive. Each one is an OS process with private memory.
  • Configuration matters. max_connections, work_mem, shared_buffers interact.
  • The bottleneck is rarely what you think. Measure, don't guess.
  • Postgres is fast enough — if you understand the constraints.
💡 From 2k to 108k TPS on the same hardware. The only thing that changed was how we talked to the database.

work_mem: spill to disk

  • Each sort/hash operation gets up to work_mem of RAM
  • Exceeds limit → spills to temp files on disk (slow)
  • Danger: work_mem × active connections × ops per query = total memory
demo_work_mem.py
# 64kB work_mem → external merge sort (disk) SET work_mem = '64kB'; EXPLAIN ANALYZE SELECT * FROM orders ORDER BY amount DESC; -- Sort Method: external merge Disk: 4096kB # 256MB work_mem → in-memory quicksort SET work_mem = '256MB'; EXPLAIN ANALYZE SELECT * FROM orders ORDER BY amount DESC; -- Sort Method: quicksort Memory: 12800kB
⚠ Setting work_mem = 256MB with 100 connections = up to 25 GB just for sorts. Same memory explosion that killed us in slide 2.
