El punto de partida 02 · 10

Cuatro preguntas que
la mayoría no puede responder.

01

¿Cuántos TPS aguanta tu Postgres antes de saturarse?

Y qué recurso se satura primero: CPU, I/O, locks, o conexiones.

02

¿Qué pagas cuando distribuyes la base de datos?

Más nodos no es más velocidad. ¿Cuánta latencia añade cada garantía?

03

¿Por qué el mismo hardware es 50x más rápido en OLAP?

No es el software — es el layout físico de los datos en disco.

04

¿Es tu sistema "tiempo real" o solo micro-batch rápido?

La diferencia importa cuando el SLA es 500ms, no 30 segundos.

Si dudaste en alguna — este curso es para ti.
Al terminar este curso 03 · 10

Sabrás hacer esto.
Con evidencia.

  • ✓

    Benchmarkear Postgres hasta el límite — identificar qué recurso se satura primero usando flame graphs y pg_stat_statements.

  • ✓

    Construir un CDC consumer desde cero con psycopg3 que mantiene una vista materializada en DuckDB sincronizada con Postgres.

  • ✓

    Medir la diferencia real entre Spark y Flink — producir CDFs de latencia p99 en workloads idénticos y recomendar con datos.

  • ✓

    Correr un pipeline end-to-end — Postgres → CDC → Kafka → Spark → ClickHouse → FastAPI, con Grafana monitoreando cada etapa.

No conclusiones abstractas — código que corre.

Perfil recomendado 08 · 10

Este curso es para ti si…

  • 🐍
    Sabes Python y SQL, y ya te sientes cómodo con async básico No necesitas ser experto, pero debes poder leer código asyncio sin que te cueste.
  • 📦
    Has usado Docker alguna vez — aunque sea para correr Postgres en local No necesitas saber escribir Dockerfiles, pero sí sentirte cómodo con docker compose up.
  • 📈
    Quieres pasar de "sé Python" a "diseño pipelines de producción" El salto de junior a senior data engineer no es escribir más código — es entender qué se rompe y por qué.

No es para ti si…

Buscas un curso de Python básico o de SQL desde cero. Este curso asume que ya escribes código — viene a enseñarte a pensar en sistemas.
